# 2026-07-22 · 胜利 = 清桌；抽牌区是工具

## 现象（玩家）

游戏区已空，抽牌区仍有牌 → 既不显示胜利，又要在库/抽出叠之间「收尾配对」，像没打完。

## 根因（设计冲突）

| 文档 / 代码 | 说法 |
|-------------|------|
| `02_game_rules` D10 | 胜利 = **只清谜题区**；库可剩余 |
| 中途实现 | `isWon` = 全牌 dead；清桌后 stock 顶可点，强迫抽光 |

配点又把 stock 垫到 ~16 张（L1 伙伴 + 偶数 + 容错），桌面自洽挖完后**必然**常剩库牌 → 伪终局。

## 设计定稿（BUG → 规则）

1. **D10 回归**：胜利 = 谜题区无存活牌  
2. **D10b**：抽牌区 / 抽出叠 = **辅助工具**，过关允许残留  
3. 清桌瞬间 **自动回收** 未用 stock/waste（`reclaimUnusedDeck`），状态干净  
4. **stock 永远不可点**；只能抽到 waste 再与桌面 free 配  
5. **否决**「清桌后库内收尾配对」

## 代码

- `rules.ts`：`isWon` / `reclaimUnusedDeck`；stock `isFree` 恒 false  
- `state.ts`：配对后若桌空 → 回收库，再 `commit`  
- `main.ts`：去掉「桌空点库 = 选牌」分支  
- 单测 / `15_level_rank_design` / `04_decisions_log` 已对齐  

## 续：残局「桌面 2 张、库还剩一摞」

**原因：** 旧 `buildPhaseStock` 把库 **pad 到 ~16**，另加 L1 伙伴；桌面主要靠自配对消时，库几乎不动，残局就显得「库比桌面还厚」。

**设计：** 库 = 工具，体积与「还需要帮桌上补的点」挂钩：

1. 奇点各 1 张（R1）  
2. 至多 2 组 L1 抽牌伙伴对（+4）  
3. **禁止** 再灌填料到固定张数  

清桌即胜 + 未用库回收；残局不应再堆无意义废对。
